Specify Setup

Specify's flexible design can accomodate a single user with a single collection or multiple collections representing multiple disciplines within an institution - all sharing the same database. This is accomplished by creating relationships among collection data as well as modeling how collections are organized within institutions themselves and then configuring components within Specify specifically for each institution level.

What this means for you, the user, is that Specify 6 can be configured to work congruently with existing collection management procedures, but it requires a thoughtful setup process. Before starting the setup process, it is important to understand the modules within Specify and the institutional level at which they are associated as this will inform decisions that need to be made during the installation and setup process (that in some cases can not be undone or reversed once data has been entered).

Specify Institution Levels

The Specify data management levels mimic the common levels of hierarchy within institutions. Access to information and Specify tools are then set at the various levels.

  • Institution
    • Division
      • Discipline
        • Collection
          • Users

 

An example of an institution hierarchy may be:

  • Natural History Museum
    • Botany
      • Vascular Plants
      • Non-vascular Plants
      • Lichens
        • Teaching Collection
        • Kansas Collection
          • John Smith

The Specify setup wizard prompts for information to add one of each of the levels. Once the wizard is complete and an empty database is created any additional levels and users can be added within Specify to create the organizational structure for the institution.

The current release of Specify allows users to be created at the collection and discipline level, but not at the division or institutional level. The ability to create users at these higher levels, with access to all collections will be available in a future release. A singe user is created during the Specify wizard process. Thereafter additional users can be created through the Specify security tools. Multiple division/discipline institutions will need to copy those users that require access to multiple divisions/disciplines across to the other divisons/disciplines.

Users are futher organizaed into Groups with a default set of priviledges and permissions associated with each group:

  • Manager
  • FullAccess
  • LimitedAccess
  • Guest

Users can be placed into a user group based on the tasks they will perform and the functionality of the modules they will require, and an institutional level that reflects the data they need to access. Permissions can be altered for individual users within these user groups to allow or deny access to various data and tools.

Each instance of Specify is bound to a collection and the data contained within that collection. In other words, a manager at a discipline level can access all the collections within the discipline, but can not access the data within all the collections during the same session. To view data in another collection the user simply needs to change to that collection.

Forms

The form system is a window to the data within Specify. Data is entered, viewed and edited via the form system. The various forms within Specify are scoped at the discipline level. In other words, when a collection is created and placed within a discipline Specify will then make the forms for that Discipline available to the collection, and all collections share the same forms. A form editor is not shipped with Specify 6, but forms can be exported as a set, edited in XML and reimported. When the form set is imported back into Specify it can be saved at a different institution level, making the new forms available at a different institution level. In other words, a collection manager could export their discipline forms, edit them and reimport them at the collection level. The edited forms would then be available to any user in the collection, but not to other collections within the discipline.

Data for collection objects is available at the collection level, although it resides in one institutional database. Much of the background, or related data may be the same throughout the discipline or division levels. It makes sense then that much of the related data is available at levels other than the collection. For instance, agents are often the same within a given division and it would not make sense for every collection to enter the same agent data; therefore the agents table is available at the division level. Trees

Specify 6 stores various table data in heirarchical Tree format. These include Taxon, Geography, Storage, Chronostratigraphy and Lithostratigraphy. Tree data for Taxon, Geography, Chronostratigraphy and Lithostratigraphy is available at the discipline level, which means that all collections within a discipline share the same tree. There is only one Storage tree, scoped to the institution.

When the tree is being edited both the form and tree will be locked to all other users within the discipline for editing purposes but available to view.

In the current release of Specify the Full Name may not be edited once nodes are active on a tree; therefore it is highly recommended that the tree definitions be completed before entering collection data into Specify. It may also be helpful to complete the trees themselves.

Reports

Reports are used to present the results of Specify searches in an attractive and reader-friendly arrangement, either on paper or on screen. Reports are generated by report definitions, which are in turn based on Specify queries.

Users have the choice of making a report available at either the user or discipline level at the time the report is saved. Reports can also be exported, much like forms, and imported back into Specify at any institution level. Reports are exported as a file that contains both the report and the query that created it. Reports can also be created in XML code and imported into Specify.

Searches

One of the benefits and goals of entering data into the Specify database is to be able to retrieve the data in an organized manner. Specify's Query builder offers a flexible interface for defining and running searches. Queries are a precise request for information retrieval and can restrict the type of returned records based on the contents of almost any of Specify's data fields.

Queries created in Specify are bound to the user, making them accessible within any collection the user may login to. Queries can also be exported and imported, making the same query available to any Specify user. The collection object data that is queried is limited to the data within the collection.

Simple Searches have been pre-configured in Specify 6 from pre-indexed fields in the database. The purpose of the simple search is to eliminate the need to create a complex Query for commonly executed searches. While the pre-indexed fields are predetermined, the simple searches can be configured to include or exclude these fields as well as many related fields.

Preferences

Preferences are kept on the local machine are are available for each user. Most of the preferences apply only to the individual user, but some refer to system links and will require information from the system administrator. For example, an attachment repository is necessary when including attachments and will need to be created by the system administrator. Users will then need to enter the link to the attachment repository in the necessary preference field.

System Configuration

The collections, disciplines and divisions for the institution are created and edited in the system configuration module. Changes within this module affect the entire institution and will be reflected in the security module and used when adding users and user permissions.

When determining the structure for an institution it is important to understand the levels at which various data types are stored. An alternate way to think about this is to identify the organizational level at which the contents of data records are unique. For example, Collection Object records are 'scoped' to the collections level, because all Collection Object records within a collection must be unique. No two records can have the exact same values. In contrast Locality records are scoped to discipline, which means they can be shared among collection databases within the same discipline, and they must be unique records within the scope of a discipline. Schema Configuration

"Schema" refers to the many tables and fields that systematize the many dimensions and relationships of the data within biological collections. The schema configuration module is a highly functional, though somewhat complex, set of tools created to configure the Specify schema for individual discipline needs. Changes made in the schema configuration tool function at the discipline level and will affect all collections within a given discipline.

Specific tasks within the schema configuration include:

  • Hiding tables and fields that are not used within a discipline.
  • Renaming field captions that are displayed within Specify.
  • Writing table and field descriptions.
  • Denoting a field as required for the purpose of data entry.
  • Creating and editing table formats and aggregates .
  • Editing system pick lists.
  • Changing a string field into a number format, pick list, or web link.

Number Scheme

Specify 6 provides a very powerful formatting and auto-numbering facility. Specify's number schemes are particularly useful for institutions that use specific formats for Catalog, Loan, Accession and/or Gift Numbers. Specify allows any string field to be formatted for a numbering scheme. A numbering scheme uses a numbering format with the possibility of participating in a number series. Number schemes are scoped to the discipline level, except for Catalog Numbers, which are bound to a number scheme at the time they are created. This was implemented so that each collection could have a unique format and/or number series. Accession Numbers that have been bound to an institution during the Wizard Setup can not be edited in the Schema Configuration.

A number format refers to the order of numbers or characters that make up the format. For instance, an institution may have a catalog number that starts with the institution code, then adds the current year and ends with an auto-incrementing number, separated by dashes. The format for that number would be KU-2009-0001. This is only one sample format. Formats can consist of alphanumeric, numeric, alphabet, year and auto-incrementing numbers. Auto-incrementing fields are joined to the auto numbering tool inside Specify. When auto numbering is turned on the number will be generated by Specify, and when auto numbering is turned off the number can be manually entered. Auto-incrementing numbers must then be joined to a number series.

The number series refers to the number that is auto-incremented, or auto-generated by Specify. The importance of the number series not only lies in the concept of auto numbering as mentioned above; it is also critical to understand which collections within the institution will need to participate in the same number series.

Pick Lists and Web Links

Specify ships with system pick lists and web links, which can be edited. New pick lists and web links can also be created, but will not automatically appear within existing forms. To add the new pick lists and web links to forms, the form set must be exported, edited to reflect the change and then imported back into Specify.

Security

Specify has a comprehensive system for adding users, then granting permission to the program's tools and to data. When Specify starts up it checks the user permissions and loads various components accordingly.  Users who are not given permission to use modules in Specify will not see the corresponding buttons that connect users to that module. For instance, users who are not given permission to edit Trees will not have a Tree button on the Taskbar or have the ability to add the Tree button in the user Preferences tool.

Currently, it is not possible to search for collection objects across collections in Specify 6 but that is a capability we are committed to in a future release. This is specifically tailored for Registrars, Directors and Conservators.

Import/Export

Both forms and reports can be exported and imported into Specify using this module. Forms can be exported, changed in XML and then reimported back into Specify. Reports can be exported then imported back into Specify at a different level; in this way reports can be shared within an institution. This ability would also allow both forms and reports to be consistent within an entire community of collectors.

Importing and exporting is based on the same levels that are found within the institution, as well as two additional levels:

  • System
    • Institution
      • Division
        • Discipline
          • Collection
            • Group
              • Users

Group refers to the same groups of users as above:

  • Manager
  • FullAccess
  • LimitedAccess
  • Guest

For each level the imported resource (form or report) is made available to other users at the same level as well as all levels below it. This is based on the permissions for the person importing. Resources imported at the bottom (User) level will be made available to the user only. At the Group level the form/report is accessible to all users within the collection that are part of the same Group. For example, if a Limited Access user imports a report from a tissue collection within the Lichen discipline of Botany into the Division level the report will be available to any user within any collection in the Lichen discipline.

Resources can not be imported or exported from the System level. It holds the resources that Specify ships with and can be used to restore these default resources if needed.

WorkBench/Uploader

The WorkBench was released in 2007 as a stand-alone application. The goal was to offer collection managers a tool to use in the field as well as tools to clean their data before uploading the data into the Specify database.

The WorkBench is included in the suite of tools for Specify 6. Currently, the Upload function is not available from the stand-alone version of the WorkBench. Data sets created in the stand-alone version will need to be exported as Excel files and imported into the Specify 6 WorkBench.

Backup/Restore

Specify ships with a separate Backup and Restore tool. Backup paths must be set for the institution based on the MySQL setup. The Backup and Restore functions are NOT limited to single collections, but backup and restore the entire database.

 

Setup Tasks

Once the Specify Database Setup Wizard has been completed a blank database will be attached to Specify. The blank database includes one institution with one division, one discipline within the division, and one collection within the discipline, as well as one user. The database will need to be expanded to include all the necessary users, collections, divisions and disciplines. Each discipline will also require to be configured according to their specific needs.
